T-050  [~] Deployment
       Decide hosting (Railway, Fly.io, and Render all work well with
       Django + SQLite-to-Postgres upgrade). Add Whitenoise for static files.
       Add a Procfile or equivalent.

       Remaining (deploy-time):
         [ ] Decide host (leaning Railway)
         [ ] Whitenoise for static files
         [ ] Procfile / start command (gunicorn)
         [ ] SQLite -> Postgres upgrade
         [ ] Set LOGTAIL_SOURCE_TOKEN + LOGTAIL_INGESTING_HOST as host vars
             (separate Better Stack source from local/dev)
         [ ] Add platform log drain for startup/crash output (see below)

       Done (app-side logging, the observability slice):
         [x] Better Stack (Logtail) app logging wired in config/settings/base.py
             — LogtailHandler behind a QueueHandler (non-blocking, avoids the
             runserver autoreloader deadlock), gated on the two env vars,
             console handler always on, timestamps forced to UTC. Verified
             logs reach Better Stack. NOTE: this is server telemetry only;
             user/domain event logging (e.g. claiming a ground) is a separate
             future task.

       Host choice — leaning Railway, but not locked in. Whichever host is
       chosen, the logging integration depends on two things:

       1. Better Stack env vars. The app ships logs via the in-app
          LogtailHandler (see config/settings/base.py LOGGING), gated on
          LOGTAIL_SOURCE_TOKEN + LOGTAIL_INGESTING_HOST. Both must be set as
          host service/environment variables in production (no .env is
          deployed). On Railway these are service Variables; the equivalent
          on Fly.io is `fly secrets set`, on Render the Environment tab.

       2. Startup / crash output. The in-app handler only captures logs that
          reach Django's logger — it does NOT capture gunicorn boot output,
          import-time errors, failed migrations, or process crashes, which go
          to stdout/stderr before Django logging is configured. To capture
          those, add a platform log drain that forwards stdout/stderr to
          Better Stack:
            - Railway: add a Better Stack log drain (Settings > log drains).
            - Fly.io: configure a log shipper (fly-log-shipper) to Better Stack.
            - Render: use the Better Stack syslog/HTTP drain.
          The in-app handler (structured app logs) and the drain (raw
          startup/crash output) are complementary — set up both for full
          coverage.

       Note when picking the host: confirm it supports an outbound log drain
       to a third party. Railway, Fly.io, and Render all do; if the final
       choice is something else, verify this before committing.
